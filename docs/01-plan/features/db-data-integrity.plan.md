## 1. 현황 분석 및 리서치

### 1.1 현재 데이터 처리 방식
- `deal_year >= 2024` 데이터 대상.
- `HAVING COUNT(*) >= 3` 기준으로 평균(`AVG`) 산출.
- 문제점: 3건 중 1건만 특수 거래(가족 간 증여성 거래, 임대)여도 평균이 크게 왜곡됨.

### 1.2 필터링 대상 정의
- **임대 단지:** LH, SH, 국민임대, 행복주택, 장기전세 등 (시세보다 현저히 낮음)
- **소규모 단지:** 거래량이 적어 통계적 대표성이 부족한 경우.
- **가격 아웃라이어:** 해당 평형대 시세에서 ±30% 이상 벗어난 극단적 거래.

## 2. 설계 (Strategy)

### 2.1 데이터 정제 알고리즘 (Data Cleaning)
1.  **Keyword Blacklist:** 단지명(`apt_name`)에 아래 키워드 포함 시 제외.
    - `임대`, `행복주택`, `LH`, `SH`, `국민주택`, `영구임대`, `도시형생활주택` (일부), `공공임대`
2.  **Volume Threshold:** 최소 거래 건수를 `5건` 이상으로 상향 (데이터 부족 시 단계적 완화).
3.  **Statistical Outlier Removal:**
    - `Trimmed Mean` 적용: 상위 10%, 하위 10% 거래를 제외한 나머지 80%의 평균 산출.
    - 또는 `Median` (중앙값) 사용 검토.

### 2.2 백엔드 구현 (FastAPI)
- `server/main.py`의 `optimize_location` 쿼리 수정.
- SQL 레벨에서 1차 필터링 후, Python 로직에서 2차 정교화 처리.

## 4. 검증 방법
- 특정 지역(예: 강남구)의 데이터 중 임대 단지가 검색 결과에서 사라졌는지 확인.
- 동일 단지의 평균가가 극단적인 거래에 의해 요동치지 않는지 테스트 데이터 대조.
