# [Plan] 백엔드 데이터 파이프라인 확장: 전월세 실거래가 연동 및 프론트엔드 반영

## 1. 개요
- **기능명**: 국토교통부 아파트 전월세 실거래가 API 연동 및 분석 모델 통합
- **목적**: 현재 매매(매수) 위주로만 계산되던 실거래가 데이터를 전세/월세 시장까지 확장하여, 실제 타겟 유저(사회초년생, 신혼부부 등)의 체감 거주비를 보다 현실적으로 산출하기 위함입니다.
- **대상 범위**: 백엔드 데이터 수집기(`collector.py`), 분석 API(`main.py`), 프론트엔드 UI(`App.jsx`)

## 2. 상세 요구사항 (요청사항 반영)

### 2.1 백엔드 데이터 파이프라인 (Data Collection)
- **API 엔드포인트 변경/추가**: 제공받은 새로운 서비스 키(`7c7b7f2b751248958a8bf6bba48481d621f202f946cf8028056a89b3c9feb532`)와 국토교통부 **아파트 전월세 실거래가 자료 API**(`http://apis.data.go.kr/1613000/RTMSDataSvcAptRent`)를 연동.
- **데이터베이스 스키마 확장**: 전월세 데이터(보증금, 월세, 전월세 구분 등)를 저장할 수 있도록 SQLite 데이터베이스(`search_house.db`) 스키마 재설계 혹은 별도 테이블(`rent_transactions`) 구축.
- **기존 매매 수집기 유지**: 기존의 매매 실거래가(`collector.py`) 코드는 삭제하지 않고 유지하되, 전월세 데이터 수집을 병렬 또는 별도 스크립트로 분리하여 관리.

### 2.2 분석 모델 고도화 (main.py)
- **전월세 거주비 계산식 도입**: 기존의 [매매가 * 0.04 / 12] 계산식 대신, 실제 수집된 **"보증금 + 월세"** 데이터를 기반으로 한 기회비용 산출 로직 적용.
  - 전세의 경우: 보증금의 기회비용(예: 예금 금리 혹은 대출 이자율 반영, 보수적으로 연 3~4%) 계산하여 월 거주비 산출.
  - 월세 및 반전세의 경우: (보증금 * 이자율 / 12) + 실제 월세.
- 반전세(보증금+월세 혼합형) 처리를 위해 전세보증금(deposit)과 월세(monthly_rent) 컬럼을 모두 파싱하여 합산 계산.

### 2.3 프론트엔드 UI/UX 변경 (임시 조치 포함)
- **전월세 전용 모드 전환**: 현재 UI에서 "매매 / 임대" 선택 버튼을 임시로 숨기거나(Hidden), 기본값을 무조건 **'전월세(임대)'** 로 고정하여 사용자 혼선 방지.
- **거주비 UI 텍스트 변경**: 결과 카드의 "추정 전세가" 혹은 "평균 매매가" 표기를 **"평균 전세 보증금"** 또는 **"평균 월세(보증금/월세)"** 형태로 직관적으로 변경.

## 3. 제약 및 고려사항 (반전세 이슈 해결 방안)
전월세 API는 전세와 월세(반전세 포함)가 혼재되어 반환됩니다. 이를 프론트엔드에 어떻게 보여줄 것인지가 핵심입니다.
*   **분석 모델 통일 방안 (전월세 전환율 적용)**: 복잡한 반전세 데이터를 유저에게 한 줄로 요약해 주기 위해, 수집된 모든 보증금과 월세를 **'전세 환산가(전세보증금으로 통일)'** 또는 **'완전 월세(모두 월세로 환산)'** 비용으로 내부적으로 통일시켜 랭킹을 매기고, UI에는 대표값(예: 최근 가장 많이 거래된 전세/월세의 원본 데이터)을 노출하는 방식을 고려합니다.
