## 1. 현황 분석 및 리서치

### 1.1 현재 알고리즘의 한계
- 현재: `dist / speed + penalty` 방식.
- 문제: 같은 10km라도 9호선 급행 지역과 버스 환승 지역의 차이를 구분하지 못함.

### 1.2 가용 API 검토
- **방법 A: Kakao Mobility API**
    - 장점: 대중교통 길찾기 데이터가 정확함.
    - 제약: API 키 발급 필요, 호출량 최적화 필수.
- **방법 B: ODsay (대중교통 전문 API)**
    - 장점: 한국 내 대중교통 데이터 특화.
- **방법 C: Naver Direction API**
    - 장점: 지도 UI와 연동이 매끄러움.

## 2. 설계 (Strategy)

### 2.1 하이브리드 알고리즘 프로세스
1.  **Step 1 (DB Search):** 연봉/주거비 비율/면적 필터를 적용하여 후보 단지 50개 추출 (직선거리 랭킹).
2.  **Step 2 (Batch Request):** 상위 20개 단지 좌표 ↔ 직장 좌표 쌍에 대해 Route API 호출.
3.  **Step 3 (Re-scoring):** API로 받은 **실제 소요 시간**을 'Fatigue Model'에 대입하여 최종 비용 산출.
4.  **Step 4 (Cache):** 한 번 계산된 역-역 간 소요 시간은 DB에 캐싱하여 API 호출 최소화.

### 2.2 호출량 최적화 (적절한 호출)
- 모든 단지가 아닌, **대표 역(Station) 간의 시간**을 먼저 계산.
- "단지 → 역(도보)"은 계산식으로, "역 → 직장역(대중교통)"은 API로 처리하여 효율 극대화.

- API 응답 속도 및 서버 부하 테스트.
